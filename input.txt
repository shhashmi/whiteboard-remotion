"### From Loop to Architecture\n\nSection 1.1 introduced the observe-think-act loop. Now we go deeper: how do you structure the loop? How do you handle complex tasks that require planning? When should one agent delegate to others?\n\nThree architectural patterns dominate production agentic systems:\n\n| Pattern | Structure | Best For | Trade-offs |\n|---------|-----------|----------|------------|\n| **ReAct** | Interleaved reason-act cycles | Exploratory tasks, debugging, research | Flexible but can drift; higher latency |\n| **Plan-and-Execute** | Full plan first, then execute | Defined multi-step workflows | Faster execution, but rigid; replanning is expensive |\n| **Hierarchical** | Supervisor delegates to specialized sub-agents | Complex domains requiring expertise | Powerful but coordination overhead |\n\n[[diagram:0]]\n\n### ReAct: The Interleaved Pattern\n\nReAct agents alternate between reasoning and acting on every cycle. The key insight: **explicit reasoning traces** let you debug agent behavior and enable the model to \"think out loud.\"\n\n```python\n# Simplified ReAct loop structure\nwhile not done:\n    # THINK: Generate reasoning about current state\n    thought = llm.generate(f\"\"\"Current state: {observation}\n    Goal: {goal}\n    What should I do next and why?\"\"\")\n    \n    # ACT: Parse thought into tool call\n    action = parse_action(thought)\n    \n    # OBSERVE: Execute and get result\n    observation = execute_tool(action)\n    \n    # Check termination\n    done = is_goal_achieved(observation)\n```\n\nLangGraph's `create_react_agent` and similar framework methods implement this pattern. The reasoning trace becomes your audit log—when the agent misbehaves, you read the thoughts to understand why.\n\n[[diagram:1]]\n\n### Plan-and-Execute: Upfront Planning\n\nPlan-and-Execute separates concerns: a **planner** generates the full task breakdown, then **executors** handle each step. This offers two advantages:\n\n1. **Cost efficiency**: The expensive reasoning model (GPT-4, Claude Opus) plans once. Cheaper models or deterministic code can execute.\n2. **Parallelization**: Independent steps can run concurrently.\n\n```\nUser: \"Set up a new microservice with auth, logging, and tests\"\n\n[PLANNER] Generates:\n  1. Create service scaffold\n  2. Add authentication middleware\n  3. Configure logging\n  4. Write unit tests\n  5. Write integration tests\n\n[EXECUTOR 1] Creates scaffold (can be deterministic)\n[EXECUTOR 2] Adds auth (may need reasoning)\n[EXECUTOR 3] Configures logging (deterministic)\n[EXECUTOR 4-5] Write tests (may parallelize)\n```\n\nThe trade-off: if the plan is wrong, you've wasted execution. ReAct can course-correct mid-task; Plan-and-Execute commits upfront.\n\n[[diagram:2]]\n\n### Hierarchical Agents: Divide and Conquer\n\nWhen tasks require different expertise domains, a single agent struggles. Hierarchical architectures solve this with **supervisor** and **worker** agents:\n\n- **Supervisor/Orchestrator**: High-level reasoning, task decomposition, delegation decisions\n- **Specialist Workers**: Domain-specific execution (code agent, search agent, database agent)\n\nGoogle's whitepaper identifies this as one of eight essential multi-agent patterns. The supervisor doesn't execute—it coordinates.\n\n```\nUser: \"Investigate this production incident\"\n\n[SUPERVISOR] Reasons: Need to check logs, metrics, and recent deployments\n├── [LOG AGENT] Searches and summarizes relevant logs\n├── [METRICS AGENT] Queries dashboards, identifies anomalies  \n└── [DEPLOY AGENT] Checks recent releases, identifies risky changes\n\n[SUPERVISOR] Synthesizes findings, identifies root cause\n```\n\nKey design decisions:\n- **Communication**: How do workers report back? Full context or summaries?\n- **Authority**: Can workers spawn sub-workers, or only the supervisor?\n- **Failure handling**: If a worker fails, does the supervisor retry, substitute, or escalate?\n\n### Self-Improvement and Reflection\n\nThe most advanced pattern: agents that **learn from their own failures**. Reflection mechanisms enable this:\n\n1. **Critic agents**: A separate agent evaluates the primary agent's output, flagging errors\n2. **Iterative refinement**: Output → Critique → Revision → Repeat until quality threshold met\n3. **Experience accumulation**: Successful trajectories become few-shot examples for future tasks\n\nResearch shows these patterns dramatically improve reliability. The SEAL approach improved factual accuracy from 33.5% to 47%. Self-generated in-context examples lifted task completion from 73% to 89% on ALFWorld benchmarks.\n\n```python\n# Generator-Critic pattern\nfor attempt in range(max_attempts):\n    output = generator_agent.generate(task)\n    critique = critic_agent.evaluate(output)\n    \n    if critique.passes_threshold:\n        return output\n    \n    task = task + f\"\\nPrevious attempt: {output}\\nCritique: {critique}\"\n```\n\nThe meta-insight: reflection costs tokens and latency, but dramatically reduces the blast radius of errors in production."
